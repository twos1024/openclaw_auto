# ClawDesk / OpenClaw 首次安装与首启 UI 设计文档

> 说明：本文件保留历史文件名以减少仓库迁移成本，但内容已改为 ClawDesk 的真实首启 UI 设计规范。
>
> 日期：2026-03-20

---

## 1. 设计目标

ClawDesk 的首启界面不是一个“泛化设置页”，而是一个明确的安装引导器。UI 需要把复杂度压到最低，让用户只做两类动作：

1. 看懂当前卡在哪一步。
2. 按当前步骤完成输入或点击下一步。

### 1.1 设计原则

1. 先结果，后细节。优先展示“还差什么才能进入 Dashboard”。
2. 单路径。首启流程只保留一条主线，不给用户多个互相竞争的入口。
3. 可恢复。任一步失败都要能原地重试。
4. 可解释。每一步都要给出状态、日志、建议。
5. 低认知负担。不要在首启阶段暴露 provider 管理、Agent 管理、Channel 管理等业务概念。

---

## 2. 与当前实现的 UI mismatch

当前仓库中仍能看到旧的 provider-first 首启痕迹，但目标 UI 应该改成：

1. 环境检查。
2. OpenClaw 安装。
3. OpenAI-compatible API 配置。
4. Gateway 安装与启动。
5. Dashboard 进入。

也就是说，首启 UI 不应先让用户创建 Provider，再把错误抛给 Gateway。首启阶段要把 API 配置收敛成单一表单，把 Gateway 作为最后的启动目标。

---

## 3. 视觉基调

### 3.1 总体风格

推荐风格：

1. 深色底色 + 高对比内容卡片。
2. 几何感强、信息块清晰。
3. 稍强的品牌识别，但不要喧宾夺主。
4. 进度与状态信息比装饰元素更重要。

### 3.2 色彩建议

| 角色 | 建议 |
|------|------|
| 主背景 | 深灰或近黑，保持沉稳和桌面工具感 |
| 内容卡片 | 比背景更亮一档，形成层次 |
| 主强调色 | 用于当前步骤、主按钮、成功状态 |
| 警告色 | 用于环境缺失、端口冲突、配置失败 |
| 文字 | 高对比白 / 灰白，不用过度装饰 |

### 3.3 字体与节奏

1. 标题使用大字号、低字重，建立“向导式”气质。
2. 正文使用更高可读性字号。
3. 所有关键状态都要通过字形、颜色、图标三重编码表达，不只依赖颜色。

## 4. 首启页面结构

### 4.1 页面框架

```
┌────────────────────────────────────────────────────────────┐
│ 顶部标题区                                                 │
│ ClawDesk 首次安装向导   当前步骤 3/5   状态标签            │
├────────────────────────────────────────────────────────────┤
│ 左侧：步骤导航 / 说明               右侧：当前步骤主内容      │
│ ┌───────────────────────┐           ┌─────────────────────┐ │
│ │ 1 欢迎                │           │ 当前步骤卡片         │ │
│ │ 2 环境检查            │           │ 表单 / 状态 / 日志   │ │
│ │ 3 安装 OpenClaw       │           │                     │ │
│ │ 4 配置 API            │           │                     │ │
│ │ 5 启动 Gateway        │           │                     │ │
│ │ 6 打开 Dashboard      │           │                     │ │
│ └───────────────────────┘           └─────────────────────┘ │
├────────────────────────────────────────────────────────────┤
│ 底部操作区：上一步 / 下一步 / 重试 / 打开 Dashboard         │
└────────────────────────────────────────────────────────────┘
```

### 4.2 布局职责

| 区域 | 职责 |
|------|------|
| 顶部标题区 | 说明当前流程与整体进度 |
| 左侧步骤区 | 作为“任务清单”，显示已完成、当前、待完成 |
| 右侧主内容区 | 展示表单、安装结果、健康检查、错误说明 |
| 底部操作区 | 只保留当前步骤需要的按钮，避免干扰 |

## 5. 步骤级 UI 设计

### 5.1 Step 1 欢迎页

内容：

1. 产品标题。
2. 一句话说明。
3. 5 步或 6 步流程概览。
4. 一个主按钮进入环境检查。

交互要求：

1. 主按钮始终明确。
2. 不展示高级设置。
3. 给出“无需手动反复确认”的提示。

### 5.2 Step 2 环境检查页

内容模块：

1. 平台摘要卡。
2. Node / npm / OpenClaw 检测卡。
3. 自动补齐状态卡。
4. 环境日志折叠区。

UI 细节：

1. 每个检测项都要显示 `已检测 / 缺失 / 正在修复 / 失败`。
2. 缺失项要显示建议动作，但默认给出自动修复按钮。
3. 自动修复中，主按钮应变成 loading 状态并锁定。

验收：

1. 用户可以一眼看出到底缺什么。
2. 修复前后状态变化必须清晰。

### 5.3 Step 3 安装 OpenClaw 页

内容模块：

1. 当前版本目标。
2. 安装进度条。
3. 分阶段 timeline。
4. 详细日志面板。
5. 完成后版本校验结果。

交互要求：

1. 安装按钮文案应是“安装 OpenClaw”或“重新安装/升级”。
2. 安装开始后不可跳步。
3. 完成后直接把用户带到 API 配置页。

状态展示：

| 状态 | UI 表达 |
|------|---------|
| 安装中 | 主按钮 loading，进度条前进 |
| 已安装 | 绿色状态，显示版本号 |
| 失败 | 红色状态，附带错误分类 |

### 5.4 Step 4 配置 OpenAI-compatible API

内容模块：

1. API Key 输入框。
2. Base URL 可选输入。
3. Model ID 可选输入。
4. 兼容模式提示。
5. 校验按钮与保存按钮。

UI 细节：

1. API Key 必须默认 masked。
2. Base URL / Model ID 使用辅助说明，不要默认隐藏到高级设置。
3. 保存前可以做“测试并保存”或“验证并保存”。
4. 错误提示应该贴近字段，而不是只在页面底部报错。

交互要求：

1. 校验失败时，字段高亮。
2. 成功后自动推进到 Gateway 步骤。
3. 不允许用户在未配置 API 的情况下进入 Gateway 启动页。

### 5.5 Step 5 安装并启动 Gateway

内容模块：

1. Gateway 状态卡。
2. 安装/注册状态。
3. 启动状态。
4. 本地地址、端口、PID 或等价状态。
5. 日志摘要。

UI 细节：

1. 需要明确显示“正在启动 Gateway”“Gateway 已就绪”“启动失败”。
2. 端口冲突时要提供可见的冲突提示。
3. 启动成功后，主按钮应变成“打开 Dashboard”。

### 5.6 Step 6 Dashboard 进入页

内容模块：

1. 成功态插画或图标。
2. Dashboard 已就绪提示。
3. 打开 Dashboard 的主按钮。
4. 可选的“回到主页”入口。

交互要求：

1. 进入页必须明确告诉用户已经完成初始化。
2. 如果内嵌 Dashboard 失败，要给外部打开入口。
3. 不要让用户回看“安装”按钮作为主要行动。

## 6. 组件规范

### 6.1 步骤导航组件

要求：

1. 显示总步数。
2. 支持已完成、当前、未开始三态。
3. 已完成项可点击回看，但不能破坏流程。

视觉规则：

1. 当前步最醒目。
2. 已完成步带勾。
3. 未完成步弱化显示。

### 6.2 状态卡组件

状态卡用于环境、安装、配置、Gateway。

字段建议：

1. 标题。
2. 主状态。
3. 辅助说明。
4. 版本 / 路径 / 地址 / 端口 / PID 等信息。
5. 重试或修复按钮。

### 6.3 日志面板

要求：

1. 默认只显示最近几行摘要。
2. 可展开查看完整日志。
3. 日志必须支持复制。
4. 敏感信息自动掩码。

### 6.4 表单组件

要求：

1. API Key 输入采用密码样式。
2. Base URL 使用普通输入框。
3. Model ID 允许用户直接输入或从下拉建议选择。
4. 校验提示直接贴近输入框。

## 7. 交互与状态

### 7.1 主按钮文案规则

| 步骤 | 主按钮 |
|------|--------|
| 欢迎 | 开始检查环境 |
| 环境 | 自动修复 / 重新检测 |
| 安装 | 安装 OpenClaw |
| 配置 | 验证并保存 |
| Gateway | 启动 Gateway |
| 完成 | 打开 Dashboard |

### 7.2 错误状态规则

1. 错误必须给出“发生了什么”和“下一步怎么做”。
2. 不允许只显示技术栈错误码。
3. 如果是可恢复问题，按钮要直接给重试。
4. 如果是用户输入问题，错误必须绑定到字段。

### 7.3 禁止状态

1. 安装进行中禁止跳到后续步骤。
2. Gateway 未就绪时禁止进入 Dashboard。
3. API Key 未保存时禁止启动 Gateway。
4. 环境未准备好时不要展示安装完成态。

## 8. 首启页面文案建议

### 8.1 首页标题

建议使用：

1. `初始化向导`
2. `首次安装`
3. `完成环境检查、安装、配置与启动`

### 8.2 关键提示文案

1. `我们会自动检查并补齐运行环境。`
2. `先安装 OpenClaw，再配置 API Key，最后启动 Gateway。`
3. `完成后将自动打开 Dashboard。`

### 8.3 错误提示文案

1. `Node 22+ 未检测到，请先完成环境补齐。`
2. `OpenClaw 安装失败，请检查网络或权限后重试。`
3. `API Key 校验失败，请确认 Base URL 和模型 ID。`
4. `Gateway 启动失败，请查看端口冲突或日志。`

## 9. 响应式与适配

### 9.1 桌面优先

首启流程以桌面端宽屏为主，建议布局如下：

| 宽度 | 布局 |
|------|------|
| >= 1280px | 左步骤导航 + 右主内容双栏 |
| 1024-1279px | 左导航收缩，主内容保留 |
| < 1024px | 单栏堆叠，步骤导航折叠 |

### 9.2 最小可用宽度

建议 1024px 作为舒适下限，低于此宽度时要自动适配为单栏，不允许内容挤压导致字段不可读。

---

## 10. 可访问性要求

1. 所有步骤按钮支持键盘聚焦。
2. 状态颜色之外必须有文本。
3. 进度变化需要 aria live 或等价提示。
4. 日志面板可被屏幕阅读器访问。
5. 输入错误必须能被辅助技术识别。

---

## 11. 与实现的映射关系

| UI 区块 | 主要数据来源 | 备注 |
|------|------|------|
| 环境检查卡 | 环境检测命令 / hook | 需要展示 Node / npm / OpenClaw |
| 安装卡 | 安装服务 / install progress | 需要进度、阶段、日志 |
| API 配置表单 | 配置服务 / form state | 保存后掩码显示 |
| Gateway 状态卡 | Gateway health / status | 需要地址、端口、PID、状态 |
| Dashboard 入口 | Gateway ready state | 仅在就绪后可点击 |

---

## 12. QA 检查清单

1. 首启是否按环境 -> 安装 -> 配置 -> Gateway -> Dashboard 的顺序进行。
2. 是否还出现 provider-first 的早期路径。
3. 缺少 Node / npm 时，界面是否能引导自动补齐。
4. 安装完成后是否显示 OpenClaw 版本。
5. API Key 输入是否被掩码处理。
6. Gateway 未就绪时 Dashboard 是否保持禁用。
7. 启动失败时是否给出可执行的下一步建议。
8. 成功后是否能自动进入 Dashboard。
