# ClawDesk v0.2.0 Command Center and Guided Runbook

## Summary

ClawDesk v0.2.0 upgrades the current desktop console from a set of isolated pages into a guided operations workspace. The release introduces a first-class Runbook view, a global runtime banner, shared page scaffolding, and clearer cross-page navigation between Install, Config, Service, Logs, Settings, and Dashboard.

The goal is to reduce user confusion during first-run setup and recovery work by making the current runtime state, active blocker, and recommended next action visible from every major page.

## Product Goals

1. Expose runtime status globally so users immediately know whether they are in browser preview, native desktop mode, or a broken desktop bridge state.
2. Turn Setup Assistant into a reusable workflow model that powers both a dialog and a full-page Runbook.
3. Standardize page-level interaction patterns so every page has a consistent hero, status summary, and action area.
4. Reduce context switching by surfacing related actions and blockers directly on the page where the user is working.
5. Preserve the current MVP boundary: ClawDesk still manages install, configure, operate, diagnose, and embedded dashboard access without re-implementing OpenClaw itself.

## Non-Goals

1. No custom chat UI.
2. No redesign of Rust command contracts unless required to support the new UX.
3. No new installer transport or package source logic.
4. No deep visual theme system or CSS framework migration.

## Scope

### 1. Global Workspace Runtime Banner

Add a shared banner under the app header that shows:

1. Current runtime mode: `live`, `browser-preview`, or `desktop-runtime-unavailable`
2. Bridge source and readiness
3. Platform and app version
4. Current primary workflow recommendation

Behavior:

1. In browser preview, the banner must clearly say the UI is read-only and local commands are unavailable.
2. In desktop runtime unavailable mode, the banner must explicitly state this is a native-shell integration problem, not a user install problem.
3. In live mode, the banner should summarize the current overall health and next action.

### 2. First-Class Runbook Page

Add a dedicated `/runbook` page and sidebar entry.

The Runbook page becomes the full-screen version of Setup Assistant and includes:

1. Overall workflow summary
2. Launch checks for install, config, service, runtime, and settings
3. Current blockers
4. Ordered setup steps with action buttons
5. Related recovery links to Logs and Settings

Behavior:

1. The Runbook must derive its state from the same overview/runtime data model used elsewhere.
2. The Runbook must never contradict the Overview page regarding current blockers.
3. The Setup Assistant dialog should reuse the same builder/model as the Runbook page, but render a compact layout.

### 3. Shared Page Scaffolding

Introduce shared layout components for:

1. Page hero
2. Surface card
3. Notice banner
4. Status badge
5. Quick action strip

Behavior:

1. Install, Logs, and Settings must be migrated to the new scaffold in this iteration.
2. Overview and Runbook should use the same visual vocabulary.
3. Inline page-specific styling should be reduced where practical, but this iteration does not require a full design-token system.

### 4. Install / Logs / Settings Workflow Enhancements

#### Install

Add a contextual workflow strip that shows:

1. Environment readiness
2. Current install phase
3. Where to go next after install completes or partially fails

#### Logs

Add a workflow-aware context card that shows:

1. Active runtime mode
2. Whether logs are expected to exist yet
3. Quick jump actions back to Service, Install, or Settings

#### Settings

Add a stronger runtime diagnostics section with:

1. Runtime mode label
2. Tauri shell detection
3. Invoke bridge detection
4. Bridge source
5. Quick actions to Runbook and Logs

## Information Architecture

### Navigation

Sidebar order:

1. Overview
2. Runbook
3. Dashboard
4. Install
5. Config
6. Service
7. Logs
8. Settings

### Setup Assistant

The top-right workspace button remains available, but the dialog becomes a compact overlay over the same Runbook model.

## Data Model Changes

Add frontend-only workspace/runbook model types to represent:

1. Workspace banner state
2. Runbook summary
3. Runbook blockers
4. Runbook support links

No Rust model changes are required for this iteration if the existing overview/runtime/service/config/settings inputs are sufficient.

## Implementation Strategy

1. Build a pure frontend service that converts `OverviewStatus + runtime diagnostics` into:
   - workspace banner model
   - runbook model
2. Add shared presentational components for banner, hero, card, badge, and quick links.
3. Add `/runbook` route and page.
4. Refactor `SetupAssistantDialog` to consume the new runbook model.
5. Apply the shared scaffold to Install, Logs, and Settings.

## Acceptance Criteria

1. A native desktop user can see a global banner that correctly distinguishes live mode from preview mode and from desktop runtime failure.
2. The new Runbook page exists and shows the same current blocker as Setup Assistant.
3. Install, Logs, and Settings all expose related next actions instead of acting as isolated pages.
4. Logs and Settings do not tell native users they are in browser preview when the real issue is runtime bridge failure.
5. The UI remains functional on Windows, macOS, and Linux because all new logic is frontend-only and depends on existing cross-platform service contracts.

## Test Plan

Tests are intentionally deferred until after implementation for this iteration, per requested workflow.

Coverage to add after implementation:

1. Unit tests for workspace/runbook model builders
2. Integration tests for page/service behavior in preview vs runtime-unavailable vs live mode
3. E2E verification that the new Runbook route and Install/Logs/Settings workflow strips render correctly
